The Problem

Analysts Don't
Need More Data.

The core insight driving this project wasn't technical — it was a product observation: the problem is not data availability. World Bank, UN FAO, OECD, and dozens of other institutions publish rich datasets. The problem is that analysts spend 60–70% of their time cleaning, joining, and normalizing data before they can answer a single research question.

The pipeline was designed around analytical questions first, not data schemas first. Instead of asking "what data do we have?", the design process started with "what questions does an economist or policy researcher actually ask?"

PM-First Design Principle

The API endpoints were designed to answer research questions, not expose raw database tables. An analyst can ask "how does GDP per capita correlate with deforestation in Sub-Saharan Africa?" — not just "give me the GDP table."

Users

  • Policy Researchers: Need cross-country comparisons, trend analysis, and anomaly detection. Time-sensitive — deadlines drive everything. Can't wait for manual data prep.
  • Economists & Analysts: Deep domain expertise, but not data engineers. Want clean, normalized, query-ready data in formats they know (CSV, JSON, SQL).
  • NGO Program Teams: Need country-level snapshots covering economic + environmental signals together — not separate databases.
  • Data Journalists: Need fast access to defensible, well-sourced numbers they can cite. Provenance matters as much as the data.
PM Thinking

Platform Product
Decisions.

Why This Is a Platform PM Problem

This isn't just a data engineering project — it's a platform product. The decisions that mattered most weren't technical (which database, which API). They were product decisions: what's the right abstraction layer for the user? What should be automatic? What should be configurable? Where does the platform end and the analyst begin?

Scope Decision: 5 Countries

Why 5 Countries, Not 50

Breadth creates maintenance debt and degrades data quality. 5 countries across 3 continents with full coverage beats 50 countries with patchy, inconsistent data. The selection covers Sub-Saharan Africa, South America, and South/Southeast Asia — maximum geographic and economic diversity at manageable depth.

API Design: Questions, Not Tables

The most important product decision was the API contract. Instead of exposing raw database tables ("GET /gdp?country=Kenya"), endpoints were designed around research questions:

GET /api/v1/correlation?metric_a=gdp_per_capita&metric_b=forest_coverage&countries=Kenya,Brazil&years=2010-2023
Returns pre-computed correlation coefficient, time-series data for both metrics, and data provenance — sourced, normalized, ready to cite.
GET /api/v1/compare?metric=renewable_energy_pct&countries=India,Denmark,Philippines&baseline_year=2015
Returns indexed comparison table (2015=100) with trend direction, outlier flags, and source documentation per data point.

Key Build vs. Buy Decisions

  • ETL layer (Build): Each API has its own schema, rate limits, and update cadence. A generic ETL framework would have added complexity without enough benefit at this scale.
  • Query layer (Build with PostgreSQL): Analytical queries are complex and require joins across tables. An in-house PostgreSQL instance gives full control. Managed DB adds cost without clear benefit for v1.
  • API wrapper (Flask): Lightweight, well-documented, Python-native. Aligns with analyst skill set; no need for heavy framework.
  • Containerization (Docker): Reproducibility is a product requirement, not just a dev practice. Analysts who clone the repo need the same environment. Docker enforces that.

Tradeoffs

  • Data freshness vs. cost: Real-time API calls on every request would hit rate limits and add latency. Solution: nightly scheduled ingestion with cache-last-fetch timestamp exposed in API response. Analysts know exactly how fresh their data is.
  • Completeness vs. reliability: Some metrics aren't available for all countries all years. Decision: never impute, never fill silently. Missing data is explicit — null with a reason code.
  • Schema flexibility vs. query speed: Normalized relational schema chosen over document store. Research queries join across metrics; relational wins for this access pattern.

Impact (Honest Framing)

  • Designed API contract around analyst research questions, not database schemas
  • Made explicit build vs. buy decisions with documented rationale
  • Chose PostgreSQL relational model over document store based on access-pattern analysis
  • Enforced data provenance — every data point carries source, last-fetch timestamp, and quality flags
  • Designed for reproducibility: Docker Compose ensures identical environment from clone to run

What This Demonstrates as a PM

  • User empathy over technical defaults: The API design started with "what does the analyst need to ask?" not "what does our schema look like?"
  • Opinionated scoping: 5 countries with full coverage beats 50 with gaps. Depth beats breadth for v1.
  • Tradeoff thinking: Every design decision (freshness vs. cost, schema vs. flexibility, build vs. buy) is a documented product tradeoff, not a technical default.
  • Platform thinking: The pipeline is infrastructure. The API is the product. Analyst experience is the design surface.
